AI Usage Documentation for DPS React Challenge

Overview
This document tracks all AI-assisted prompts and decisions made during the development of the German Address Validator application.


Prompts Used

1. Initial Project Setup
Prompt:
"I need to build a German address validator using React, Vite, and the Open PLZ API. The form should have two fields: Locality and Postal Code. 
When a user types a locality, it should auto-fill the postal code if there's only one, or show a dropdown if there are multiple. 
When a user enters a postal code, it should auto-fill the locality. Include 1-second debounce on inputs. Can you help me structure this project?"

AI Assistance:
- Initially generated the code base with cursor prompt. I copied a prompt template from cursor to serve as a guide for generating the folder structure.
- Suggested project structure with separate folders for components, hooks, and services
- Recommended creating a custom useDebounce hook
- Advised on API service layer pattern

Code Generated:
- Basic folder structure
- Initial component scaffolding


2. API Integration
Prompt:
"How do I integrate with the Open PLZ API (https://openplzapi.org/de)? I need to search localities by name and postal codes by code. 
What endpoints should I use and how should I handle the responses?"

AI Assistance:
- Explained the API endpoints: `/Localities?name=` and `/Localities?postalCode=`
- Suggested error handling patterns
- Recommended TypeScript interfaces for type safety

Code Generated:
- `openPlzApi.ts` service file with searchLocalitiesByName and searchPostalCodesByCode functions
- TypeScript interfaces for Locality and PostalCode

Modifications Made:
- Added additional response format handling (array vs object with results/data properties)
- Added validation for 5-digit postal code format
- Enhanced error handling with console logging

Code Snippets Rejected

Rejected Snippet 1: Using axios instead of fetch
Reason: Wanted to keep dependencies minimal and fetch is native

Rejected Snippet 2: Using React Hook Form
Reason: Form is simple enough that useState is sufficient

Rejected Snippet 3: Separate components for input fields
Reason: Over-engineering for this small form; kept it in one component


Decision-Making Process

1. API Integration: Chose to use the Localities endpoint for both searches because it provides more consistent data structure
2. State Management: Decided against Redux/Context as the state is localized to one component
3. Styling: Used Tailwind CSS for rapid development and easy dark mode support
4. Debouncing: Implemented custom hook instead of using lodash to reduce bundle size
5. Error Handling: Chose user-friendly messages over technical error details
